Thoughts on Progressive Enhancement

Any given browser's job is to fetch web pages and parse the associated HTML document. The traditional web stack the browser handles is HTML for semantic content, CSS for design, and JS for dynamic (programmatic) behavior. Not all browsers are created equally, and even browsers of the same type might not be configured the same way. One browser might only support plaintext, while another user has JS disabled on theirs. This issue is related to web accessibility, the idea that users, regardless of their device or browser, should enjoy a similar and thorough website experience as was intended by the developers. The question to solve is: As a web developer, how do you design your website to be compatible across different client endpoints (those with limited functionality or high-end)? Two relevant solutions (really, philosophies that guide dev workflow) are graceful degradation and progressive enhancement. Both assume an endpoint disparity but operate inversely relative to one another. Graceful degradation is reactive. Websites are designed for the most powerful, functional endpoints. If deployed to a weak client, the higher-level features are stripped away until the site reaches a compatible functionality layer. This layer may not be optimized for the best possible UX it can provide. With the traditional web stack, this means designing a lofty website with nice CSS/JS, deploying it, and degrading if a client can't process one of these layers. Depending on the endpoint, the site could be reduced to plaintext. On the other hand, progressive enhancement is proactive and works by designing each content layer from the ground up. First, the dev focuses only on the content and semantic markup (HTML). This should be designed to operate sufficiently without upper layers. Then, design-time factors such as external CSS (linked outside of the HTML doc) are layered on top of this markup. Then, dynamic behavior can be added through unobtrusive JS (the underlying HTML/CSS is not written for the JS). The key idea is that the underlying layers are not reliant on the ones above. The HTML isn't dependent on whether the browser can load/apply the CSS/JS properly. In this way, if the browser can't process a higher layer, it doesn't matter too much; the available layers are enough to provide at least a decent UX. Both workflows guarantee a universal baseline of accessibility. However, progressive enhancement more strongly emphasizes the base layer and is thus more likely to provide a more enjoyable baseline UX. However, more careful testing is required when considering how higher layers interact with lower ones. The reason this issue of web accessibility matters is that the general population is heavily dependent on the web as a primary source of their information. If we expect websites to work at their absolute peak and don't build up our stack assuming the worst-case, I personally think we are bound to give users on low-quality browsers/devices a worse experience. I believe information is a universal right, and what your hardware/software is limited to shouldn't too strongly limit what information you can learn and access. Something as simple as how you build your website can affect your users' education, situational awareness, and overall, how they perceive their reality. From another personal perspective, building web layers from the ground up makes more sense to me. To me, centralizing assets and documents is tricky and leads to messy work.
